# Post-92 Pilot Conversion And Technical Proof Design

Current as of 2026-06-15.

## Goal

Prepare the next queued phase after `86–92` by defining how NEXUS should become easier to pilot, easier to prove, and more technically convincing without broadening beyond the current three-outage wedge.

## Selected Approach

Chosen approach: **pilot conversion first, technical proof second**

This keeps the next phase aligned to commercial readiness instead of adding more outage breadth too early.

## Why This Approach

If `86–92` lands well, the product will already have:

- three coherent outage families
- bounded reproduction and debugging for the wedge
- stronger LLM-shaped triage
- stronger agent visibility

The next most important question will be:

**can this survive a real pilot and hold up under engineering review?**

That is why this phase prioritizes:

- tenant repeatability
- pilot-proof durability
- delivery closure
- operator ease of use
- engineering trust in the handoff
- fresh-incident consistency

## Scope

### Included

- tenant onboarding v2 and pilot setup kit
- pilot ROI instrumentation and case-based proof capture
- delivery and feedback closure v2
- operator workflow simplification and role clarity v2
- repo-aware debugger handoff deepening
- runtime evidence weighting and mitigation confidence deepening
- fresh-incident triage quality calibration and evaluation

### Not included

- many new outage families
- universal debugging claims
- arbitrary reproduction claims
- platform broadening

## Product Truth Model

This phase must keep the same strict distinction between:

- inferred reasoning
- memory-backed reasoning
- runtime-backed reasoning

It must also avoid implying repo awareness equals full repository debugging coverage.

## Acceptance Criteria

The phase is complete only when:

1. pilot setup is more repeatable
2. pilot-value proof is more durable and case-based
3. delivery and outcome capture are more complete
4. operator repeated-use friction is lower
5. engineering handoff is more actionable
6. runtime evidence drives recommendations more clearly
7. fresh-incident quality is more stable and more transparently bounded
8. the phase closes with a clean checkpoint
